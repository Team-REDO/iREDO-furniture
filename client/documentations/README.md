# iREDO Furniture – Frontend Client

The client-side application for the iREDO used furniture marketplace. Built with React and TypeScript, it communicates exclusively with the API gateway and never directly with individual microservices.

---

## Tech Stack

| Technology | Role |
|---|---|
| [Vite](https://vitejs.dev/) | Build tool and dev server |
| [React 19](https://react.dev/) | UI framework |
| [TypeScript](https://www.typescriptlang.org/) | Type safety across the codebase |
| [TanStack Router](https://tanstack.com/router) | File-based, fully type-safe routing |
| [TanStack Query](https://tanstack.com/query) | Server state management, caching, and data fetching |
| [Axios](https://axios-http.com/) | HTTP client for API gateway communication |

## Getting Started

### Prerequisites

- Node.js 18+
- npm 9+
- The API gateway running locally (see root README)

### Installation

```bash
# From the monorepo root
cd client

# Install dependencies
npm install

# Copy the environment variable template
cp .env.example .env
```

Fill in your `.env` values (see [Environment Variables](#environment-variables) below), then start the dev server:

```bash
npm run dev
```

The app will be available at `http://localhost:5173`.

## Environment Variables

Create a `.env` file in the `client/` directory based on `.env.example`:

```env
VITE_API_GATEWAY_URL=http://localhost:3000
VITE_GOOGLE_CLIENT_ID=your-google-oauth-client-id
```

> **Important:** Vite only exposes variables prefixed with `VITE_` to the browser. Do not put secrets here. The root-level `.env` in the monorepo is for Docker/service configuration — keep client-specific variables in `client/.env`.

## Architecture & Key Concepts

### Communication with the Backend

The client **only communicates with the API gateway** — never directly with individual microservices. All requests go through the single `apiClient` defined in `src/api/client.ts`.

```
Browser → API Gateway (Express) → Microservices
```

### API Client (`src/api/client.ts`)

The Axios instance handles two things automatically for every request:

- **Auth token injection** – attaches the Google OAuth `id_token` as a `Bearer` token in the `Authorization` header
- **401 handling** – redirects to the login page globally if the token is expired or invalid

If you need to call a new endpoint, import `apiClient` from here — do not create separate Axios instances.

### Routing (TanStack Router)

Routes are file-based under `src/routes/`. Adding a new file automatically registers a new route — the `routeTree.gen.ts` file is regenerated by the Vite plugin on save. **Do not manually edit `routeTree.gen.ts`.**

Route parameters are fully type-safe. For example, `/listings/:id` means `params.id` is typed and validated automatically.

### Data Fetching (TanStack Query)

TanStack Query manages all server state — caching, background refetching, loading/error states, and mutations. The general pattern is:

1. Define a query function that calls `apiClient`
2. Use `useQuery` or `useMutation` in your component
3. For route-level prefetching, use route loaders that populate the Query cache before the component renders

The Query client is passed through the Router context so loaders can prefetch data.

### Authentication (Google OAuth)

Authentication is handled via Google OAuth 2.0. After a successful sign-in, the `id_token` is stored and attached to all API requests via the Axios interceptor. Authorization (what a user is allowed to do) is enforced by the API gateway and microservices — the client only passes the token along.

## Development Notes

- **Devtools:** TanStack Router Devtools and TanStack Query Devtools are enabled in development. They appear in the browser and let you inspect routes, params, cache state, and query status. They are automatically excluded from production builds.
- **Adding a new route:** Create a new `.tsx` file under `src/routes/`. Follow the existing file naming conventions (`$param` for dynamic segments, `index.tsx` for index routes).
- **Adding a new API call:** Add a typed function in `src/api/` that uses `apiClient`, then consume it with `useQuery` or `useMutation`.
- **Environment issues:** If the app can't reach the gateway, check that `VITE_API_GATEWAY_URL` in `.env` matches the port the gateway is running on.
